Phase 2: Client Action & Automated Monitoring

After initiation, the system communicates with the client to gather what's needed. This phase is a dialogue between our automated system and the client, ensuring all prerequisites are met before the final transfer.

1. System Notifies Client

An automated email is sent requesting the client to perform necessary actions at their current registrar (e.g., unlock domains, disable privacy).

2. Client Provides EPP Codes

The client is directed to a secure portal or asked to reply with the EPP/Authorization codes for their batch of domains.

3. Handling NS Changes (If Applicable)

If nameservers need to change, the system requests the zone file from the client. Our CS team then uses this information to populate the zone on our platform, ensuring a seamless transition.

4. System Monitors & Executes

Once all prerequisites are met (including NS propagation if needed), our system automatically executes the final registrar transfer for the entire batch.

Stage 2 In-Depth: Build AI Agents & Rules Engine

This internal, non-client-facing stage focuses on building the core intelligence that will automate the complex rules for ccTLD transfers.

üïµÔ∏è

1. Develop the AI "Scout" Agent (Data Gathering)

Objective: To automatically and continuously monitor official sources for any changes to domain transfer policies.

  • Identify Sources: Compile and maintain a list of official registry websites for each ccTLD you plan to support (e.g., CIRA for .ca, DENIC for .de).
  • Schedule Monitoring: Implement a scheduled task (e.g., weekly) for the Scout agent to visit these key policy pages.
  • Detect & Collect: The agent will detect changes to text or documents and collect the new raw data (HTML, text, or PDF content).
üß†

2. Develop the AI "Analyst" Agent (Information Extraction)

Objective: To read complex policy documents and turn them into simple, structured data.

  • Prompt Engineering: Design a specific prompt template for an LLM to act as a policy expert and extract key information.
  • Data Extraction: The Analyst will process raw data to extract specific fields like Authorization Method, Residency Requirement, and Required Documents.
  • Structure Output: The extracted information will be formatted into a clean, machine-readable format (like JSON) for the Rules Engine.
üõ°Ô∏è

3. Build ccTLD Rules Engine & Validation Dashboard

Objective: To store the structured rules and create the "human-in-the-loop" workflow for your team.

  • Create Database: Design and build the database to store the structured rule data for each ccTLD.
  • Develop "Gatekeeper" Logic: Implement the system that compares newly extracted rules with existing rules in your database.
  • Build CS Dashboard: Create an internal UI for your CS team to review, approve, or reject potential rule changes flagged by the AI.
